# План следующего сеанса: шесть направлений улучшений

Документ зафиксирован для пошаговой работы. После каждого пункта имеет смысл прогонять `npm run verify:ci` и при необходимости `npm run sync:docs`.

**Порядок внедрения (рекомендуемый):** сначала то, что не ломает геймплей и расширяет настройки (2 → 4 → 5), затем метрики и шаринг (1 → 6), затем более рискованный UX жестов (3).

---

## 1. Ежедневный результат и шаринг без сервера

**Цель:** усилить daily-режим: игрок видит «лучший за сегодня» забег и может поделиться текстом (или намёком на скриншот).

**Поведение:**

- Ключ дня совпадает с уже используемой датой для `runMode === "daily"` (локальный календарь).
- В `localStorage` хранить объект вида `{ dateKey, bestTimeMs, bestScrap, runsToday }` обновлять по окончании забега **только** если активен daily и исход — победа (опционально: отдельно учитывать поражения «попыток за день» без перезаписи best).
- На карточке итога забега (run summary) при daily: блок «Сегодняшний рекорд» или «Новый рекорд дня!».
- Кнопка «Поделиться» формирует `text/plain` через `navigator.share` (мобильный) или копирование в буфер / `mailto:` fallback как в баг-репорте.

**Файлы (ориентир):** `web/game-config.js` (ключ метрик/префикс), `web/game.js` (`finishRunAsWin` / `openRunSummary`, хелперы даты), `web/locale-bundles.js`, `web/styles.css` при необходимости.

**Краевые случаи:** смена календарной даты в полночь; восстановление из автосейва не должно путать «сегодня»; RU/EN тексты шаблона шаринга.

**Критерий готовности:** после победы в daily виден рекорд дня; шаринг отдаёт осмысленную строку без ошибок в консоли.

## 2. Доступность и пресет «высокий контракт»

**Цель:** отдельная тема или флаг «high contrast» + опционально «крупные символы на клетках».

**Поведение:**

- В настройках: чекбокс или селект «Контраст / доступность» (например `accessibility: default | highContrast` или расширить `theme`).
- CSS: новые переменные или `html[data-a11y="high"]` с усиленными границами плиток, фоном, разделителями; враг/босс/выход — не только цветом (иконка в `tileSymbol` или data-атрибут + псевдоэлемент).
- Учесть `prefers-reduced-motion` (уже частично есть) — не добавлять обязательную анимацию для распознавания.

**Файлы:** `web/game-config.js`, `web/game.js` (`normalizeSettings`, `applySettingsToUi`, `tileClass` / `tileSymbol`), `web/locale-bundles.js`, `web/styles.css`.

**Критерий готовности:** переключение мгновенно видно на сетке и панели; тип клетки различим без цвета (текст/символ).

## 3. Жесты: long-press превью и уточнение тапа

**Цель:** на маленьком экране меньше ошибочных ходов и понятнее «что будет, если».

**Поведение:**

- Long-press (≈400–500 ms) на клетке с врагом рядом с игроком: тултип / мини-модалка «Урон: X, HP врага: Y» (данные из `state`).
- Long-press на выходе: «Нужно встать рядом» / «Можно зайти» по факту `manhattan`.
- Опционально второй этап: свайп с текущей клетки игрока в направлении как альтернатива второму тапу — аккуратно не конфликтовать с существующим свайпом рывка; возможно только когда рывок не вооружён.
- Таймеры `pointerdown` / `pointerup` / `pointercancel`; на iOS проверить `touch-action` и задержку 300 ms.

**Файлы:** `web/game.js` (обработчики на контейнере `#grid` или делегирование после `render`), `web/styles.css` (тултип), `web/locale-bundles.js`.

**Краевые случаи:** мультитач; открыта модалка реликвии; игра окончена; reduced motion не обязателен, но анимация тултипа — лёгкая.

**Критерий готовности:** превью не ломает обычный тап и рывок; нет утечек слушателей при `render()`.

## 4. Мини-энциклопедия (реликвии, враги, босс)

**Цель:** экран «Справочник» из существующих данных конфига и i18n.

**Поведение:**

- Новая кнопка в меню или секция в «Как играть»: список реликвий из `relicPool` + локализованные имена из `locale-bundles` (`relics.*`).
- Блок «Враги»: базовые правила (урон, HP шаблоны) — данные из кода генерации / пресетов; если числа зависят от сложности, показать таблично «легко / норм / сложно» или взять текущий пресет и пометить «для вашей сложности».
- Босс: отдельный подпункт (награда трофеев, поведение).

**Файлы:** `web/game.js` (`getInfoScreens` или новая функция `openEncyclopediaModal`), `web/index.html` (кнопка), `web/locale-bundles.js` (заголовки секций), `web/styles.css` (списки/аккордеон опционально).

**Критерий готовности:** все реликвии из пула видны; текст не дублирует длинные простыни — скролл внутри модалки как у инфо-экранов.

## 5. Стинг этажа (аудио + короткий визуал)

**Цель:** узнаваемый вход на станцию без тяжёлых ассетов.

**Поведение:**

- При `spawnFloor` / переходе на этаж: короткий звук (можно переиспользовать `transition` с разной громкостью или добавить 2–3 секундных клипа в `assets/sfx/` по номеру этажа).
- Визуал: 200–400 ms подсветка названия станции в логе или маленький бейдж под `floor` в topbar (`floorStations` из конфига).
- Уважать настройку звука и `prefers-reduced-motion` для визуала.

**Файлы:** `web/game.js` (`advanceToNextFloor`, `spawnFloor`, возможно `playSfx` варианты), `web/game-config.js` (привязка `floor → soundId` опционально), `web/locale-bundles.js`, новые файлы в `web/assets/sfx/` при добавлении звуков.

**Критерий готовности:** на каждом этаже ощущается отличие; при выключенном звуке остаётся визуальный стинг.

## 6. Серия побед и «неделя daily»

**Цель:** мягкая удерживающая метрика + новые бейджи в статистике.

**Поведение:**

- Метрики: `winStreak` (сброс при поражении), `dailyRunsThisWeek` или счётчик уникальных дат с завершённым daily за последние 7 дней (UTC или локаль — зафиксировать в коде и в описании).
- Достижения (расширить `defaultMetrics.achievements`): например `win_streak_3`, `daily_three_week`, без лишней логики в бою — только в `finishRunAsWin` / `finishRunAsDefeat`.
- Чипы в модалке статистики как у существующих достижений.

**Файлы:** `web/game-config.js`, `web/game.js` (`patchMetrics`, `unlockAchievement`, `finishRunAsWin`, `finishRunAsDefeat`), `web/locale-bundles.js`, при необходимости подсказка в run summary.

**Краевые случаи:** первая игра без метрик (миграция из старого сохранения); daily vs random (решить: серия только для random, для daily отдельный трек — явно описать в UI одной фразой).

**Критерий готовности:** стрика видно в метриках/ачивках; смена недели не ломает сохранение.

## Чеклист сеанса (копипаст)

1. Прочитать этот файл, выбрать подпункты порядка.
2. Ветка или прямой коммит на `main` — по твоему процессу.
3. После каждого блока: `npm run verify:ci`, ручной смок (Safari + при возможности Capacitor).
4. Обновить `assetVersion` в `package.json`, `npm run sync:docs`, закоммитить.
5. Отметить внизу этого файла (или в коммите) какие пункты закрыты.
